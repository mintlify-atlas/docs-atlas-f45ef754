# Quickstart Guide

This guide walks you through the complete process of exploiting your MediaTek device using Kamakiri, from initial connection to successful bootloader unlock and custom recovery installation.

<Warning>
  **Read Before Proceeding**: This process will void your warranty and may permanently damage your device if not followed correctly. Ensure you have backups of important data and understand the risks involved.
</Warning>

## Prerequisites Check

Before starting, verify you have:

- Amazon Fire TV Stick 4K (Mantis) - Device type ID: `AKPGW064GI9HE`
- USB to TTL serial adapter (3.3V)
- Soldering equipment and skills
- Linux machine with Python 3.x installed
- Required Python packages: `pyserial`, `pyusb`
- Root/sudo access on your Linux machine

<Note>
  **ModemManager Conflict**: You must disable or uninstall ModemManager before proceeding. The exploit will automatically check for this and exit if ModemManager is running.
</Note>

## Step-by-Step Walkthrough

<Steps>
  <Step title="Disable ModemManager">
    ModemManager interferes with serial communication. Disable it before starting:

    ```bash
    sudo systemctl stop ModemManager
    sudo systemctl disable ModemManager
    ```

    Alternatively, uninstall it completely:

    ```bash
    sudo apt remove modemmanager
    ```
  </Step>

  <Step title="Prepare the Device">
    1. Open your Fire TV Stick 4K and locate the serial pins
    2. Solder wires to TX, RX, and GND pins
    3. Connect to your USB-TTL adapter:
       - Device TX → Adapter RX
       - Device RX → Adapter TX  
       - Device GND → Adapter GND
    4. Connect the adapter to your Linux machine

    <Warning>
      Use a 3.3V adapter only. 5V will damage your device.
    </Warning>
  </Step>

  <Step title="Verify Serial Connection">
    Check that your serial adapter is detected:

    ```bash
    ls -l /dev/ttyUSB*
    ```

    Expected output:
    ```
    crw-rw---- 1 root dialout 188, 0 Mar 13 10:30 /dev/ttyUSB0
    ```

    Add your user to the dialout group for serial access:

    ```bash
    sudo usermod -a -G dialout $USER
    # Log out and log back in for changes to take effect
    ```
  </Step>

  <Step title="Enter Bootrom Mode">
    Power off the device completely, then power it on while holding the bootrom entry point. The device should appear on the serial port.

    Kamakiri will automatically detect the device:

    ```bash
    cd kamakiri
    ./bootrom-step.sh
    ```

    You'll see:
    ```

    [2026-03-13 10:31:47.234567] Found port = /dev/ttyUSB0
    ```

  </Step>

    With the device in fastboot mode, flash TWRP recovery:

    2. Reboot the device into recovery mode

    </Tip>

    In TWRP, you can:
    - Backup your current system
    - Flash custom ROMs
    - Install Magisk for root access
    - Modify system partitions
  </Step>

## Understanding What Happened

The bootrom exploit performed these operations:

<AccordionGroup>
  <Accordion title="1. Device Identification" icon="id-card">
    The exploit verified your device is a supported Mantis (Fire TV Stick 4K) by reading the `device_type_id` from IDME (ID Management Engine):

    ```python
    device_type_id = dev.idme_read(b"device_type_id")
    # Returns: AKPGW064GI9HE
    ```
  </Accordion>

  <Accordion title="2. GPT Validation" icon="table">
    The tool parsed the GUID Partition Table (GPT) to locate critical partitions:

    - `lk`: Little Kernel bootloader
    - `tee1`: Trusted Execution Environment  
    - `boot`: Android boot image
    - `recovery`: Recovery partition
    - `MISC`: Miscellaneous partition for boot flags

    If your GPT is corrupted, use `./gpt-fix.sh` to restore it.
  </Accordion>

  <Accordion title="3. RPMB Downgrade" icon="shield-halved">
    The Replay Protected Memory Block (RPMB) stores anti-rollback counters. The exploit:

    1. Reads the current RPMB data
    2. Zeros out the RPMB (256 bytes)
    3. Verifies the downgrade succeeded

    This allows installation of older bootloader versions that don't enforce security checks.

    ```python
    dev.rpmb_write(b"\x00" * 0x100)
    rpmb = dev.rpmb_read()
    # Verify rpmb == b"\x00" * 0x100
    ```
  </Accordion>

  <Accordion title="4. Bootloader Downgrade" icon="download">
    The exploit flashes vulnerable versions of:

    - **TrustZone (TZ)**: ARM secure world OS
    - **Little Kernel (LK)**: Primary bootloader  
    - **Preloader**: First-stage bootloader

    These older versions have security vulnerabilities that enable persistent access.
  </Accordion>

  <Accordion title="5. Payload Installation" icon="code">
    Three payloads are installed:

    - **lk-payload**: Hooks into Little Kernel for boot control
    - **microloader**: Small bootloader stub for boot redirection
    - **Preloader**: Modified preloader with security checks disabled

    These payloads work together to maintain bootloader unlock state.
  </Accordion>

  <Accordion title="6. Fastboot Flag" icon="flag">
    The exploit writes a magic string to the MISC partition to force fastboot mode on next boot:

    ```python
    block[0:16] = "FASTBOOT_PLEASE\x00".encode("utf-8")
    dev.emmc_write(gpt["MISC"][0], bytes(block))
    ```

    This ensures the device boots into fastboot after the exploit completes.
  </Accordion>
</AccordionGroup>

## Common Issues and Solutions

<AccordionGroup>
  <Accordion title="Error: ModemManager is running" icon="circle-exclamation">
    **Problem**: ModemManager interferes with serial communication.

    **Solution**:
    ```bash
    sudo systemctl stop ModemManager
    sudo systemctl disable ModemManager
    # OR
    sudo apt remove modemmanager
    ```
  </Accordion>

  <Accordion title="Error: Device not found" icon="usb">
    **Problem**: Serial adapter not detected or wrong permissions.

    **Solutions**:
    1. Check the serial device exists:
       ```bash
       ls -l /dev/ttyUSB*
       ```

    2. Add user to dialout group:
       ```bash
       sudo usermod -a -G dialout $USER
       ```

    3. Check connections (TX/RX might be swapped)

    4. Try a different USB port
  </Accordion>

  <Accordion title="Error: Serial protocol mismatch" icon="exchange">
    **Problem**: Communication error with the device.

    **Solution**:
    1. Check baud rate is correct (115200)
    2. Verify TX and RX are not swapped
    3. Ensure adapter is 3.3V (not 5V)
    4. Try power cycling the device
  </Accordion>

  <Accordion title="Error: Wrong device detected" icon="device">
    **Problem**: Device is not a supported Mantis Fire TV Stick.

    **Solution**: This exploit only supports Fire TV Stick 4K (Mantis) with device type ID `AKPGW064GI9HE`. Other devices are not supported.
  </Accordion>

  <Accordion title="Error: bad gpt" icon="table">
    **Problem**: GPT is corrupted or invalid.

    **Solution**:
    ```bash
    ./gpt-fix.sh
    ```

    This will restore the stock GPT for Mantis devices.
  </Accordion>

  <Accordion title="Error: rpmb downgrade failure" icon="shield">
    **Problem**: Failed to zero out RPMB.

    **Solution**: The exploit will automatically reboot and retry. If it fails repeatedly:
    1. Ensure good serial connection
    2. Check power supply is stable
    3. Try the exploit again from the beginning
  </Accordion>

  <Accordion title="Device boots to black screen" icon="tv">
    **Problem**: Boot process is stuck or bootloader is misconfigured.

    **Solutions**:
    1. Force fastboot mode:
       ```bash
       ./boot-fastboot.sh
       ```

    2. Force recovery mode:
       ```bash
       ./boot-recovery.sh
       ```

    3. Re-run the bootrom exploit
  </Accordion>

  <Accordion title="Fastboot not detected after reboot" icon="bolt">
    **Problem**: Device didn't enter fastboot mode.

    **Solutions**:
    1. Check USB cable and port
    2. Try:
       ```bash
       sudo fastboot devices
       ```

    3. Manually force fastboot:
       ```bash
       ./boot-fastboot.sh
       ```

    4. On some systems, you may need to install udev rules for MediaTek devices
  </Accordion>
</AccordionGroup>
